# Prompt: local LLM stack installer for macOS

Numbered so sections can be amended or appended without rewriting. Add new requirements under the matching section, or under §12.

---

## 1. Objective

Write a single, shareable shell script that provisions a private, self-hosted LLM stack on macOS, plus a GitHub-style `README.md` documenting it.

## 2. Target environment

- macOS, Apple Silicon (arm64) only. Refuse to run elsewhere, in preflight, with a clear message.
- zsh as the user shell.
- Assume a fresh OS install. Assume no console login — the machine is administered over SSH.
- Make no assumption about username or home directory path. Use `$HOME` and `whoami` throughout.
- Contain nothing specific to my network. Anything site-specific must be a flag or config value.

## 3. Deliverables

1. `llmstack-macos.sh` — one file, executable, no companion scripts.
2. `README.md` — GitHub conventions: contents list, tables, fenced blocks with language hints.
3. A model catalogue file, generated by the script on first run if absent, never overwritten afterwards.

## 4. Stack

| Component | Purpose |
|---|---|
| Ollama | inference engine |
| Open WebUI | web front-end |
| SearXNG | private web search for Open WebUI |
| Draw Things | image/video generation, optional |

Install Homebrew, Python 3.11, and a container runtime only as needed. Prefer pip/venv over containers for Open WebUI.

## 5. Service management

- Ollama and Open WebUI run as **system LaunchDaemons** in the `system` domain, so they start at boot with no login session.
- Privilege-drop both to the invoking user via `UserName`. Never run them as root.
- **Auto-login must never be required, suggested, or enabled.** This is non-negotiable; design around it.
- Validate every generated plist with `plutil -lint` before loading, and fail loudly if malformed.
- Set `HOME` explicitly in each daemon's environment. Daemons do not inherit it reliably.

## 6. Modes and CLI

Modes: `--install` (default), `--update` (alias `--upgrade`), `--status`, `--recommend`, `--uninstall`, `--help`.

Options: `--searxng-url URL`, `--searxng-port PORT`, `--webui-port PORT`, `--model TAG`, `--no-model`, `--no-drawthings`.

- `--help` is comprehensive: synopsis, every mode and option, components, file layout, ports, startup behaviour *and its limits*, requirements, post-install steps, exit status, examples.
- `--recommend` and `--status` are read-only. They must install, download, and change nothing.
- `--uninstall` confirms **each artifact separately**. Every prompt defaults to no; bare Enter skips. Destroying user data takes two confirmations.
- `--uninstall` never removes shared dependencies (Homebrew, Python, `mas`). Say so in the summary.
- `--update` backs up user data before touching anything, and restarts the stack even if an upgrade step fails.

## 7. Model selection

- Detect chip, tier (Base/Pro/Max/Ultra), memory, and free disk.
- Budget roughly 70% of unified memory for weights.
- Select against a catalogue on **two independent gates**: size must fit the budget, *and* the entry's declared minimum machine class must be met. Fitting and running well are different things — a dense 70B fits a 64 GB budget and performs badly on Pro-tier bandwidth.
- Prefer MoE over dense where both fit. Generation is memory-bandwidth-bound; MoE activates a fraction of parameters per token.
- Fall back to a lighter tier rather than recommending nothing.
- Warn when free disk is short of the model size plus headroom.

## 8. Catalogue file

- Plain text, pipe-delimited, `#` comments, greppable and hand-editable.
- Fields: minimum RAM, tag, size, architecture, role, verified flag, notes.
- Header carries `# Last-Updated: YYYY-MM-DD`.
- The script parses that date and grades staleness: recent under 90 days, worth reviewing at 90–180, very likely stale beyond 180. Report this in `--update` and `--recommend`.
- Mark each tag verified or unverified. **Mark a tag verified only if you actually confirmed it against the Ollama registry in this session.** Warn before pulling anything unverified; on pull failure, point at the library and continue rather than aborting.

## 9. Platform constraints to handle, not discover

macOS container runtimes require an active GUI login session. Docker Desktop is a GUI app and cannot be launched over SSH; Colima manages a per-user VM and is unsupported as a system daemon. Therefore:

- Use Colima, not Docker Desktop, when a runtime is needed.
- Local SearXNG cannot survive a reboot without a console login. State this plainly in `--help`, the README, and `llmstatus` output. Do not paper over it.
- Offer `--searxng-url` to delegate search to a Linux host where Docker is a real systemd service, and skip installing any runtime in that mode.

## 10. Failure modes to pre-empt

Each of these has bitten a working deployment. Handle them in code, not in prose.

1. `WEBUI_SECRET_KEY` must be hex only. Base64 values containing `+` or `=` are silently dropped from a plist environment. Generate with `secrets.token_hex(16)`, persist to a `0600` file, reuse on re-runs.
2. Set `SoftResourceLimits` and `HardResourceLimits` to 65536 for Open WebUI, or web search dies with `Too many open files`.
3. Pin `DATA_DIR` explicitly and consistently. A mismatch silently creates an empty database and looks exactly like wiped credentials.
4. Docker creates a **directory** at any bind-mount path that doesn't yet exist. Detect and remove a stray `settings.yml` directory, and write the config file before starting the container.
5. The SearXNG container listens on 8080 internally regardless of its config. Map host-port to 8080.
6. `launchctl bootstrap` fails with error 5 when already loaded. Always `bootout` first, wait, then bootstrap, and treat a bootstrap error as non-fatal after verifying with `launchctl print`.
7. Check port availability before binding; report the conflicting holder.
8. SearXNG needs `json` in its `formats` list or Open WebUI receives nothing.
9. A shell block appended to `.zshrc` on every re-run collides with itself. Guard on **every** marker string the tooling has ever written, not just the current one, and *replace* the block rather than leaving an existing one in place. Definitions of the same name that changed form between versions — an alias becoming a function — are fatal in zsh, not merely redundant: zsh expands a live alias while parsing a function of that name and the parse fails. Warn about definitions found outside any marked block instead of silently duplicating them, and tell the user to `exec zsh`, since sourcing cannot clear what is already in the running shell.

## 11. Script quality

- `set -euo pipefail`. Fully idempotent: every step checks state before acting; existing data is never overwritten.
- Poll for readiness against real endpoints. No fixed `sleep` as a substitute for a check.
- Long or large operations print progress; downloads state expected size up front.
- Never `source` an interactive rc file from inside the script.
- Delimit any `.zshrc` block with **both** start and end markers, keep a list of markers used by previous versions, and remove every match before writing a fresh block. Back the file up first. Match markers literally, not as regexes — marker text containing `(`, `)`, or `.` breaks a `sed` address.
- Never edit a user's shell config destructively without a backup, and never delete a range whose extent you cannot determine — report it and skip.
- Shell functions read settings from a config file at call time rather than having values baked into `.zshrc`.
- Comment *why*, not *what*, especially at each workaround from §10.

## 12. Documentation

`README.md` must include: what it installs, requirements, quick start, modes and options tables, how model selection works and why bandwidth matters, the catalogue format and date mechanism, the reboot limitation with its cause, remote-SearXNG setup including a working Compose file, shell commands, updating, uninstalling, file layout, ports, a troubleshooting section covering every item in §10 with the diagnostic command for each, an FAQ, and design notes explaining the contested choices.

## 13. Working style

- State assumptions and design decisions before writing code. Where options exist, name them, weigh the tradeoffs, and justify the choice.
- Push back on anything that adds risk or unnecessary complexity rather than complying silently.
- Flag uncertainty rather than hiding it. Distinguish verified fact from inference.
- Verify current facts — package names, image tags, model tags, API endpoints — rather than relying on recall. Say which ones you verified and which you didn't.
- Syntax-check the script and test any non-trivial parsing logic against representative inputs before delivering.
- Deliver complete files, never diffs or fragments.
